Compression Approaches Compared

Empirical analysis of 6 context compression strategies on 98 real compact events across 20 Claude Code sessions. We simulate what each approach would preserve or destroy — and whether the next user question needed that data.

1. Study Design

When Claude Code's context window fills up (~167k tokens), auto-compaction fires — summarizing the entire conversation into ~4k tokens. But what if we used a different approach? Would the user's next question still be answerable?

Methodology

  1. Parse 20 real JSONL sessions (25-103 MB each)
  2. Find all 98 compact_boundary events
  3. For each compact, extract the full context window that existed before compaction
  4. Simulate 6 compression approaches on that window
  5. Extract the next real user message after compaction
  6. Analyze: would each approach preserve enough context for that question?

Methodology Limitations

  • Simulation, not real testing. We simulate compression by calculating token counts. We don't actually run each approach and measure Claude's response quality. Real information loss may differ from our estimates.
  • Single developer's sessions. All 20 sessions are from one user. Usage patterns, question styles, and context dependencies may not generalize.
  • Token estimation. We use chars/4 as proxy. Our estimated pre-compact tokens often differ from Claude's reported preTokens (we capture the JSONL content, not what Claude actually sees after microcompaction).
  • "Next question" heuristic. We extract the first user message after compact that isn't a summary. Some sessions have long autonomous Claude runs before the next user input.

How to read this chart

Token Reduction % = how many tokens are removed from the context window. Higher = more space freed. 97.9% means only 2.1% of the original context remains.

Context Recall = how much useful information survives compression and is available when the user asks the next question. A user asking "что там было с таблицами ранее?" needs old context — if it was deleted, recall is low. A user asking "давай дальше" doesn't need old context — any recall level works.

The ideal approach has high token reduction AND high context recall — remove the junk (base64 noise, DOM wrappers) but keep the meaning (text, summaries, recent data).

Key Finding: The Pareto Frontier

Element-Level compression achieves 71% average reduction with LOW information loss. Full Compact achieves 98% but destroys everything. The gap between 71% and 98% is largely conversation metadata (user/assistant text, thinking) that Element-Level intentionally preserves.

In screenshot-heavy sessions (4M+ tokens pre-compact), Element-Level achieves 97-99% reduction — matching Full Compact — because screenshots dominate the token count and OCR compresses them 300-500x.

Caveat

  • Element-Level's 0% minimum occurs in pure-coding sessions with no screenshots and no large tool_results. For these sessions, it provides almost no benefit — observation masking or compact is better.
  • Observation Masking beats Element-Level in 78 of 98 events by pure token count. It's more aggressive (replaces ALL old outputs vs. only big ones). The question is whether that extra aggression loses important data.

4. Microcompaction's Fatal Flaw

Claude's built-in microcompaction (Layer 1) replaces old text tool_results with file path references. But it does not handle screenshots.

31 of 98 compacts: Microcompaction achieves less than 5% reduction

These are the screenshot-heavy sessions. Microcompaction dutifully replaces old Read and Bash outputs with path references — saving maybe 50 tokens per element — while 17 screenshots at 250k tokens each sit untouched in context.

Average screenshots per compact window: 4.0. Total screenshot tokens across all events: 63.9 million.

Microcompaction works for:

  • Pure coding sessions (file reads → path refs)
  • Sessions with many small tool_results
  • Gradual context growth over 100+ messages

Microcompaction fails for:

  • Any session with screenshots (Playwright, user-uploaded)
  • Sessions with large DOM snapshots
  • Sessions where 3-5 elements consume 90%+ of context

Surprisingly Low Re-read Rate

Only 3.1% of compacts led to immediate file re-reading. This suggests Claude's compact summaries usually preserve enough working state for the immediate next step. However:

  • We only detect explicit Read/Grep/Glob tool calls. Claude may silently produce worse code without re-reading.
  • 27% of post-compact questions are simple continuations ("продолжай", "давай дальше") — these don't test context retention.
  • The real cost of compaction may be subtle quality degradation that's hard to detect: wrong variable names, forgotten constraints, repeated mistakes.

The Sweet Spot

Element-Level compression occupies the best position: 71% average reduction (97-99% in screenshot-heavy sessions) with ~80% information fidelity. It achieves this by being selective — compressing only what's compressible (screenshots, DOM, large text) while preserving conversation structure and small elements.

Full Compact is the most space-efficient (98% reduction) but at maximum information loss. The question is: is the extra 27% reduction worth losing all original context?

8. Recommendations

Tiered Strategy (Best of All Worlds)

No single approach is optimal for all situations. The data suggests a tiered strategy:

  1. Always: OCR all screenshots immediately (even recent ones). Free, local, 300x compression, near-full context recall.
  2. At 70% context: Clean DOM snapshots (regex, 5-10x). Summarize large text outputs older than 5 messages.
  3. At 85% context: Apply observation masking to remaining old tool_results.
  4. At 95% context: Fall back to full compact as emergency measure.

This layered approach would delay or prevent full compact in most sessions, preserving significantly more context fidelity.

Why This May Not Work

  • Complexity cost. A tiered system with 4 stages, OCR, regex, AI summarization, and masking is significantly more complex than Claude's simple compact. More complexity = more failure modes.
  • 50% of questions don't need old context. Half the time, full compact works fine. The elaborate preservation system benefits only the 11% of questions that reference earlier data.
  • OCR doesn't exist yet as an MCP feature. This analysis assumes OCR works perfectly. In practice, OCR errors, non-text screenshots, and cross-platform issues (macOS vs Linux) will reduce the real-world benefit.
  • Anthropic is evolving compaction. Claude Code's compaction quality improves with each version. The 3.1% re-read rate and 0% clarification rate suggest compact is already quite good for most use cases.
  • The real benchmark is task completion. We measure token reduction and information preservation, but the real question is: does the user finish their task? That requires a different study design.
