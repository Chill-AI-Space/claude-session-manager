1. Goal & Scope

One-sentence goal

Build an npm package that acts as a transparent MCP proxy — wrapping any existing MCP server and compressing its tool results before Claude sees them — to reduce context window consumption by 60-95% while preserving high context recall.

In scope (v1.0)

  • MCP proxy server using stdio transport (same as wrapped server)
  • Screenshot OCR: base64 PNG/JPEG → extracted text via Apple Vision (macOS) / tesseract (cross-platform)
  • DOM snapshot cleanup: strip [ref=XXX], generic ARIA attributes, redundant whitespace
  • Large text truncation with configurable max tokens
  • Per-tool-name rules (e.g. "always OCR for browser_take_screenshot")
  • Passthrough mode for small results (below threshold)
  • CLI: npx context-trash-mcp --wrap "command args"
  • Zero-config defaults that work out of the box for Playwright MCP

Out of scope (v1.0, candidates for v2)

  • Gemini/LLM-powered summarization (Tier 3) — future add-on
  • HTTP/SSE transport — stdio only in v1
  • Custom compressor plugin API
  • Token counting (use byte-based heuristic: bytes / 4)
  • Dashboard / UI for compression stats

Key Principles

  • Transparent proxy — Claude doesn't know the proxy exists. Tools list, tool calls, and notifications pass through unchanged. Only CallToolResult content gets compressed.
  • Content-aware routing — each content block in a result is classified (image, DOM snapshot, large text, small text) and routed to the appropriate compressor.
  • Fail-safe — if any compressor fails, the original content passes through unmodified. Never lose data due to proxy errors.
  • Zero latency for small results — results under threshold (default: 500 tokens ≈ 2KB) pass through without processing.

Config file: config.json

  "ocrEngine": "auto",
  "verbose": false,

  "rules": [
    {
      "toolName": "browser_take_screenshot",
      "strategy": "ocr",
      "comment": "Always OCR screenshots — 300x compression"
    },
    {
      "toolName": "browser_snapshot",
      "strategy": "dom-cleanup",
      "comment": "Strip ref attributes and generic ARIA from accessibility tree"
    },
    {
      "toolName": "Read",
      "strategy": "truncate",
      "maxTokens": 3000,
      "comment": "Large file reads get truncated"
    },
    {
      "toolNamePattern": ".*",
      "strategy": "auto",
      "comment": "Default: auto-detect content type and compress accordingly"
    }
  ]
}

Strategy types

StrategyWhen to useWhat it does
ocrImage content (base64)Extract text via Vision/tesseract, replace image with text
dom-cleanupPlaywright accessibility snapshotsStrip [ref=N], generic ARIA, collapse whitespace
truncateLarge text blocksKeep first + last N tokens, add "[truncated]" marker
autoDefaultDetect content type → route to appropriate strategy
passthroughExclude specific toolsNo compression, pass as-is

Rule matching

Rules are evaluated top-to-bottom. First match wins. toolName is exact match, toolNamePattern is regex. If no rule matches, auto strategy is used.

5. Compression Pipeline

Per-request flow (on tools/call response)

1. Receive CallToolResult from wrapped server
2. Estimate token count (bytes / 4)
3. If total tokens < threshold → passthrough (return as-is)
4. Find matching rule for tool name
5. For each content block in result:
   a. Classify: image | dom-snapshot | text
   b. Apply strategy from matched rule (or auto-detect)
   c. If compressor fails → keep original block
6. Reassemble result with compressed blocks
7. Log stats to stderr (if verbose):
   "[context-trash] browser_take_screenshot: 247,000 → 823 tokens (99.7% reduction)"
8. Return compressed result to Claude

Content classification (auto strategy)

SignalClassified asCompressor
Content type = imageImageOCR
Contains [ref= or - role: patternsDOM Snapshotdom-cleanup
Text content > maxTextTokensLarge texttruncate
Text content ≤ thresholdSmall textpassthrough

6. Compressor Modules

6.1. OCR Compressor $0 P0

Input

MCP content block: { type: "image", data: "base64...", mimeType: "image/png" }

Output

MCP content block: { type: "text", text: "[OCR extracted from screenshot]\n..." }

Implementation

  1. Decode base64 → write to temp file (/tmp/ctx-trash-XXXX.png)
  2. Run OCR engine:
    • macOS (default): swift script using Apple Vision framework — best quality, free
    • Fallback: tesseract <file> stdout — cross-platform, free
  3. Delete temp file
  4. Return text block with [OCR extracted from screenshot] prefix

Apple Vision Swift script (embedded)

// Bundled as src/ocr/vision.swift, compiled on first run
import Vision
import AppKit

let url = URL(fileURLWithPath: CommandLine.arguments[1])
let image = NSImage(contentsOf: url)!
let cgImage = image.cgImage(forProposedRect: nil, context: nil, hints: nil)!

let request = VNRecognizeTextRequest()
request.recognitionLevel = .accurate
request.usesLanguageCorrection = true

let handler = VNImageRequestHandler(cgImage: cgImage)
try handler.perform([request])

let results = request.results ?? []
for observation in results {
    if let candidate = observation.topCandidates(1).first {
        print(candidate.string)
    }
}

Expected compression

Screenshot ~250,000 tokens → OCR text ~500-2,000 tokens. 99%+ reduction.

6.2. DOM Cleanup Compressor $0 P0

Input

Text block containing Playwright accessibility tree snapshot

Rules (applied sequentially)

  1. Strip ref attributes: [ref=123] → remove (Claude doesn't need element refs after snapshot)
  2. Strip generic ARIA: remove role="generic", role="none"
  3. Collapse empty nodes: remove lines with only - role: generic and no children
  4. Collapse whitespace: multiple blank lines → single blank line
  5. Deduplicate navigation: if identical nav block repeats, keep first + add [repeated nav block omitted]

Expected compression

DOM snapshot ~15,000 tokens → cleaned ~5,000-8,000 tokens. 40-65% reduction.

6.3. Text Truncation Compressor $0 P1

Strategy

Smart truncation: keep first 70% and last 30% of the max token budget, insert a [... truncated N tokens ...] marker between.

Rationale

Start of files contains imports/structure. End often contains the relevant function or error. Middle is usually the least important.

Config

{
  "maxTokens": 2000,      // total token budget
  "headRatio": 0.7,        // 70% from start
  "tailRatio": 0.3         // 30% from end
}

8. Implementation Phases

Phase 1: Transparent Proxy P0 — MVP

Goal: Working MCP proxy that passes everything through unchanged.

  • Parse CLI args (--wrap)
  • Spawn wrapped server as child process (stdio)
  • Bidirectional message forwarding: stdin → child stdin, child stdout → stdout
  • Handle process lifecycle (exit, signals, errors)
  • Verify: npx context-trash-mcp --wrap "npx @anthropic-ai/mcp-playwright" works identically to bare Playwright MCP

Test: Run with Claude Code, use Playwright tools — everything should work as before.

Phase 2: OCR Compressor P0 — Core Value

Goal: Screenshots automatically converted to text. This alone is 80% of the value.

  • Intercept CallToolResult messages
  • Detect image content blocks
  • Apple Vision OCR on macOS (compile Swift script on first run, cache binary)
  • Tesseract fallback for Linux/Windows
  • Replace image block with text block
  • Log compression stats to stderr

Test: browser_take_screenshot returns text instead of image. Claude can still "see" page content.

Phase 3: DOM Cleanup + Config P0

Goal: Accessibility tree snapshots cleaned up. Config system in place.

  • DOM cleanup compressor: regex-based rules
  • Config file loading with defaults
  • Rule matching system (toolName → strategy)
  • Content classifier (auto strategy)
  • Threshold-based passthrough

Test: browser_snapshot returns cleaned DOM. Config overrides work.

Phase 4: Text Truncation + Polish P1

Goal: Handle large text results. Publish to npm.

  • Smart truncation compressor (head + tail)
  • --dry-run mode
  • --verbose with structured stats
  • README with examples
  • npm publish as context-trash-mcp

Phase 5 (Future): LLM Summarization P2

  • Optional Gemini Flash integration for intelligent summarization
  • Strategy: summarize — send content to Gemini with "summarize for an AI coding assistant" prompt
  • Requires API key in config
  • Budget tracking per session

11. Risks & Mitigations

RiskImpactMitigation
OCR quality too low for Claude to act on High — Claude may misunderstand page content Apple Vision has excellent quality. Test on real screenshots. Add [OCR may contain errors] prefix. Allow passthrough override per tool.
DOM cleanup removes useful refs Medium — Claude can't click elements Only strip refs from snapshot tool, not from interaction tools. Playwright uses refs only during the same turn.
Proxy breaks MCP protocol High — tools stop working Phase 1 is pure passthrough. Validate against MCP spec. Integration tests with real servers.
tesseract not installed on user machine Medium — OCR fails on Linux Clear error message: "Install tesseract for OCR: brew install tesseract / apt install tesseract-ocr". Fallback to passthrough.
stdio buffering issues Medium — messages get stuck Use line-buffered JSON-RPC. Flush after each message. Test with large payloads.
Swift compilation fails Low — macOS only Pre-compile on first run, cache binary. Fall back to tesseract if swiftc missing.

12. Technical Decisions

Why stdio proxy (not HTTP)?

  • Claude Code uses stdio for local MCP servers
  • Simpler: just spawn child process and pipe
  • No port conflicts, no auth needed
  • Most MCP servers (Playwright, filesystem) are stdio

Why not use @modelcontextprotocol/sdk for the proxy?

  • SDK creates a full server + client pair — overkill for a proxy
  • Raw JSON-RPC line protocol is simple: read line → parse → maybe transform → write line
  • Fewer dependencies = faster npx startup
  • BUT: evaluate during Phase 1 — if raw protocol is too fragile, switch to SDK

Why compile Swift on first run?

  • Can't ship a binary in npm package (architecture-dependent)
  • macOS always has swiftc available
  • One-time cost: ~2 seconds to compile, then cached
  • Result: native Vision framework access without external dependencies

Token estimation: bytes / 4

  • Claude uses BPE tokenizer where avg token ≈ 4 bytes for English text
  • For images (base64), this overestimates — but that's fine (we want to compress images aggressively)
  • Good enough for threshold decisions. No need for real tokenizer dependency.
