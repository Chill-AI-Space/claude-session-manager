1. The Problem: Quadratic Token Growth

Claude Code operates within a 200,000 token context window. Every turn, the entire conversation history is re-sent to the model. This means total cumulative token usage grows quadratically — each new message costs all previous tokens plus the new one.

The N² Problem

A session with 100 messages, each averaging 2,000 tokens, doesn't cost 200k tokens. It costs:

Turn 1:   2k tokens
Turn 2:   4k tokens
Turn 3:   6k tokens
...
Turn 100: 200k tokens
Total:    ~10M tokens (not 200k!)

This is why sessions show 48M+ cumulative input tokens — the N² effect of replaying full context each turn.

When the context window fills up (~167k tokens, or ~83%), Claude Code triggers auto-compaction — summarizing the entire conversation into a compressed "working state." While this preserves continuity, it:

  • Loses fine-grained detail from earlier in the conversation
  • Costs additional tokens for the summarization itself
  • Can fail entirely on very large contexts ("Conversation too long" error)
  • Replaces structured conversation with a monolithic summary blob

External context injections — TeamHub (1,300-6,700 tokens per message), MCP tool descriptions (up to 51k tokens), system prompts, CLAUDE.md files — compound the problem by consuming fixed context on every turn.

Limitations of this framing

  • N² is theoretical maximum. In practice, caching (prompt caching, cache_read_input_tokens) significantly reduces the actual API cost. Anthropic charges much less for cached tokens. The N² problem is real for context window pressure but overstates the cost impact.
  • Context window sizes are growing. Claude's window is already 200k; competitors offer 1M+ (Gemini) and 128k+ (GPT-4). The "window full" problem may be partially solved by larger windows, though N² cost still applies.
  • Compaction isn't always bad. We frame it negatively, but for many coding sessions compaction works well enough. The quality concerns are most acute for complex multi-file refactors and browser automation — not all usage.

Analysis Pipeline

For each session JSONL, we parsed every line and extracted individual content blocks:

  • tool_result blocks (text content, base64 images, structured data)
  • tool_use blocks (input parameters to tools)
  • user_text blocks (human messages, compact summaries)
  • assistant_text blocks (Claude's text responses)
  • thinking blocks (internal reasoning)
  • image_data blocks (user-uploaded images)

Each block was sized by character count / 4 as a token estimate. We then computed distribution across size buckets, type breakdowns, and per-compact-window analysis.

Methodology Limitations

  • Small sample size. 15 sessions from a single developer. Usage patterns, tool preferences, and project types heavily reflect one person's workflow. A Playwright-heavy user will naturally produce Playwright-heavy data. Results may not generalize to the broader Claude Code population.
  • Selection bias. We analyzed our own sessions, which were selected for being "interesting" (long, many compacts). Shorter, simpler sessions that never hit context limits are underrepresented. This inflates the apparent severity of the problem.
  • Token estimation. We use chars / 4 as a token estimate. Real tokenization varies by content type — base64 is particularly inefficient (closer to chars/3), while natural language text may be closer to chars/4.5. Our estimates have ~15-25% error margin.
  • No public JSONL benchmarks. We couldn't find public raw JSONL session files for comparison. Community shares HTML transcripts (simonw), not raw data. This means we can't validate our distribution findings against other users' real sessions.
  • Temporal snapshot. Claude Code's compaction, microcompaction, and Tool Search features are evolving rapidly. Findings from Dec 2025 — Mar 2026 may already be partially outdated by newer versions.
  • Playwright bias. 13 of 15 sessions involved Playwright. This heavily skews the "screenshots dominate everything" conclusion. For the broader population (most users don't use Playwright MCP), the distribution likely looks more like our "Pure Coding" pattern.

Findings Caveats

  • "tool_result = 90-98%" is the Playwright effect. This headline number comes from browser sessions. In pure coding sessions the split is closer to 60/40. Presenting 90-98% as the general finding is misleading without the Playwright context.
  • Token count ≠ information value. A 250k-token screenshot may carry 500 tokens of useful information, but it also carries layout, color, and spatial data that text-only OCR cannot capture. For visual debugging (CSS, responsive layout, design review), this data matters.
  • Compact summaries are double-counted. We count compact summary blocks (4-6k tokens) as part of the total. These are themselves compression artifacts. Including them inflates the apparent token waste.
  • JSONL ≠ context window. We analyze the full JSONL file, which includes messages from before compaction. But Claude only sees post-compact content. The "what's in context" analysis is more relevant than "what's in JSONL" totals.

4. Two Distinct Usage Patterns

Pattern A: Browser/Playwright Sessions

Characteristics:

  • Screenshots = 90-98% of all tokens
  • Single screenshot: 150k-420k tokens
  • Compact triggers every 25-40 messages
  • Context fills in minutes of active browsing
  • Multiple compacts per session (up to 16)

Root cause: Playwright MCP returns screenshots as base64-encoded PNGs embedded directly in tool_result. A single full-page screenshot can exceed the entire 200k window.

Known issue: GitHub #21230 — "Context limit reached with only 25% context usage when taking screenshot via Playwright MCP"

Pattern B: Pure Coding Sessions

Characteristics:

  • tool_results = 60% (file reads, grep output)
  • Max single element: ~8k tokens
  • Context fills gradually over 100+ messages
  • Rare compacts (every 2-3 hours)
  • More even token distribution

Root cause: Many medium-sized elements (source files, search results) accumulate steadily. No single catastrophic element.

Implication: Different compression strategies needed — cannot just target "fat" elements.

Pattern Classification Limitations

  • Binary split is an oversimplification. Real sessions often mix both patterns — a coding session that opens a browser once, a Playwright session that does heavy file editing. We forced a binary categorization.
  • Missing pattern: API-heavy sessions. Users who call external APIs (WebFetch, custom MCP servers for databases, cloud APIs) may exhibit a third pattern with many medium-large JSON responses. We had no such sessions in our dataset.
  • Missing pattern: multi-agent / subagent sessions. Sessions using Agent tool extensively generate nested conversation blobs. We didn't analyze this pattern separately.
  • Playwright MCP is evolving. Newer versions may return smaller snapshots or support resolution control for screenshots. The 150k-420k per screenshot figure may decrease as the tool matures.

5. Existing Approaches & Literature Review

5.1. Claude Code's Built-in Compaction (3 Layers)

Layer 1: Microcompaction

Claude Code divides tool outputs into a "hot tail" (recent, fully visible) and "cold storage" (older outputs replaced with file path references). Applies to: Read, Bash, Grep, Glob, WebSearch, WebFetch, Edit, Write.

Source: Decode Claude — Inside Claude Code's Compaction System

Layer 2: Auto-compaction

Triggers at ~95% context usage (our data shows it fires at ~167k tokens, which is 83.5% of 200k). Produces a structured "working state" summary with: Task Overview, Current State, files modified, key outputs, identifiers, error messages, and a continuation instruction.

The summary is injected as a compact_boundary system message + a isCompactSummary: true user message. The JSONL is not modified — old messages remain but Claude only sees post-compact content.

Source: Anthropic API Docs — Compaction

Layer 3: Manual /compact

User-triggered via /compact command. Same mechanism as auto but can be invoked earlier. Can fail with "Conversation too long" error if context is already at limit (GitHub #13929).

Critique of Built-in Compaction Analysis

  • We lack internal data. Our understanding of compaction is based on JSONL artifacts, reverse engineering, and blog posts — not Anthropic's internal documentation. The actual algorithm may be more sophisticated than we describe.
  • Compaction quality is improving. Anthropic actively develops compaction. The quality complaints from community (section 6) may be from older versions. We didn't control for Claude Code version in our analysis.
  • Microcompaction is hard to measure. We can see compact_boundary events in JSONL, but microcompaction (hot tail / cold storage) happens transparently. We may be underestimating how much built-in compression already helps.

5.2. Academic Research

SWE-Pruner: Self-Adaptive Context Pruning for Coding Agents

arXiv 2601.16746 Jan 2026

A 0.6B parameter neural "skimmer" model performs task-aware pruning of code context. The agent formulates an explicit goal ("focus on error handling") as a hint to guide pruning targets.

Results:

  • 23-54% token reduction on SWE-bench Verified
  • Up to 14.84x compression on single-turn tasks
  • Success rate maintained or slightly improved (not degraded)
  • Interaction rounds decrease by 18-26%
  • <100ms latency even at 8K tokens

Source: arxiv.org/abs/2601.16746

The Complexity Trap (JetBrains Research / NeurIPS 2025)

arXiv 2508.21433 NeurIPS DL4Code 2025

Key finding: Simple observation masking (replacing old tool outputs with placeholders) is as effective as LLM-based summarization, while being cheaper and simpler.

Results on SWE-bench Verified:

  • Observation masking: 50%+ cost reduction, solve rate maintained
  • With Qwen3-Coder 480B: masking improved solve rate by 2.6% vs raw agent
  • In 4/5 setups, masking had the lowest cost per instance
  • A hybrid approach (masking + summarization) reduces costs by another 7-11%

Critical insight for our approach: "A typical SE agent's turn heavily skews towards observation, making observation masking the most efficient strategy." This directly validates focusing compression on tool_result blocks.

Source: arxiv.org/abs/2508.21433

Prompt Compression Survey (NAACL 2025)

NAACL 2025 Selected Oral

Comprehensive survey of prompt compression techniques including SelectiveContext, LLMLingua (Microsoft), and Nano-Capsulator. Key methods achieve 5-20x compression with 70-94% cost savings and minimal performance loss.

Source: aclanthology.org — Prompt Compression Survey

Recurrent Context Compression (ICLR 2025)

Achieves 32x compression by training a recurrent module that compresses context into fixed-size representations. While impressive, requires model fine-tuning — not applicable to closed-source API models like Claude.

Source: openreview.net/forum?id=GYk0thSY1M

Academic Literature Critique

  • SWE-bench ≠ real usage. All papers benchmark on SWE-bench (single-issue bug fixes). Real Claude Code sessions involve multi-step workflows, browser automation, and creative tasks that benchmarks don't capture. Compression that works for SWE-bench may not work for a 3-hour debugging session.
  • Observation masking has limits. JetBrains' finding that masking = summarization sounds great, but their test was on short, focused bug fixes. In long sessions where earlier outputs are referenced later (e.g., "remember that error from 20 messages ago?"), masking destroys needed context.
  • No Playwright/browser studies. None of the academic papers study browser automation sessions with screenshots. All focus on code-only agents. The most token-intensive real-world pattern has zero academic coverage.
  • Compression surveys are model-agnostic. NAACL 2025 survey covers techniques designed for open-source models with fine-tuning access. For closed-source API users (Claude, GPT), many techniques are inapplicable. We may be overstating the solution space.

5.3. The Two Distinct Problems: Input Bloat vs Output Bloat

Context window waste in Claude Code comes from two fundamentally different sources, and most existing solutions address only one of them.

Problem A: Input Bloat (Tool Definitions)

Every connected MCP server dumps its full tool schema — names, descriptions, JSON parameters, types, constraints — into the context window before your first message.

  • 81 tools = 143k tokens (72% of 200k) consumed before you type anything
  • Grows with number of connected MCP servers
  • Static cost, same every turn

Status: Largely solved by Tool Search (Claude Code v2.20+) and tool consolidation strategies.

Problem B: Output Bloat (Tool Results)

Tool outputs — screenshots, DOM snapshots, file contents, API responses — accumulate in the conversation history and are re-sent every turn.

  • Single screenshot = 150-420k tokens of base64
  • Our data: tool_result = 60-98% of all context tokens
  • Grows with conversation length (N² cumulative)

Status: Largely unsolved. Built-in compaction is the only defense, and it's lossy.

The community and industry have produced many tools — but most target Problem A (input bloat), while Problem B (output bloat) remains the dominant consumer of context in real sessions.

5.4. Solutions Targeting Input Bloat (Problem A)

Claude Code Tool Search (v2.20+)

Built-in

Claude Code reduced MCP definition bloat by 46.9% (51k → 8.5k tokens) via deferred tool loading. Tools are loaded on-demand instead of all at startup. Auto-activates when tool descriptions exceed 10% of context window.

Targets: Problem A only. Does not affect tool result sizes.

Source: Medium — Claude Code Just Cut MCP Context Bloat

mcp-compressor (Atlassian Labs)

Open Source Atlassian Labs

A proxy MCP server that wraps other MCP servers and compresses tool descriptions into a two-step interface. Instead of loading all parameters at once, the agent first sees a slim summary and requests full details only when needed.

Targets: Problem A only. Tool results pass through unmodified.

Speakeasy Dynamic Toolsets

Commercial speakeasy.com

Reduces input tokens by 96% by dynamically loading only relevant tool definitions. Makes it practical to build MCP servers with hundreds of tools without overwhelming context.

Targets: Problem A only.

ToolHive MCP Optimizer (Stacklok)

Open Source Stacklok / Dev.to

Containerized MCP server management with token optimization. Reduces the overhead of having many MCP servers connected simultaneously.

Targets: Problem A primarily.

McPick / Context Isolation

Community

Manual approaches: toggle MCP servers on/off before sessions (McPick), or use slash commands to spawn isolated Claude instances with separate MCP configs (paddo.dev).

Targets: Problem A. Requires user discipline.

5.5. Solutions Targeting Output Bloat (Problem B)

claude-context-mode (MCP Server)

Open Source github.com/mksglu/claude-context-mode | Author's blog post

What it is: An MCP server that gives Claude a separate "sandbox" to run code in. Instead of Claude calling Bash/Read/Grep directly (where the full output lands in context), Claude calls context_mode.execute("git log --oneline -50") — the output stays in the sandbox, and only a small summary returns to Claude's context window.

How it works step-by-step:

1. Claude wants to see git log (normally: 50k tokens of output)
   │
2. Instead of Bash tool, Claude calls:
   context_mode.execute({ code: "git log --oneline -50", lang: "shell" })
   │
3. Context Mode spawns an ISOLATED subprocess:
   ┌──────────────────────────────┐
   │  Sandbox (separate process)  │
   │  - runs: git log --oneline   │
   │  - captures: full stdout     │
   │  - stores: in sandbox memory │
   │  - returns: only summary     │
   └──────────────────────────────┘
   │
4. Claude receives: "50 commits found. Latest: abc1234 fix: navbar"
   (~100 tokens instead of ~50,000)
   │
5. If Claude needs details, it can:
   context_mode.execute({ code: "git show abc1234", lang: "shell" })
   (another sandbox call, another small summary)

Key technical details:

  • 10 language runtimes: JS, TS, Python, Shell, Ruby, Go, Rust, PHP, Perl, R — Claude can execute code in any of these inside the sandbox
  • Process isolation: Each execute spawns a separate subprocess. Scripts can't access each other's memory.
  • Auth passthrough: gh, aws, gcloud, kubectl, docker credentials are inherited from the host environment
  • Knowledge base: SQLite FTS5 index with BM25 ranking. Claude can index docs and search them without loading full content into context.
  • Permission inheritance: If sudo is blocked in Claude Code, it's also blocked in the sandbox

Results: 315 KB → 5.4 KB (98% reduction). Sessions that hit slowdown at 30 min now run 3+ hours (6x longer). Git analysis on MCP servers repo: 99.7% reduction.

Critical difference from other approaches: Context Mode doesn't compress existing tool outputs — it replaces the workflow entirely. Claude must learn to use context_mode.execute() instead of Bash/Read/Grep. This is a paradigm shift, not a transparent proxy.

Limitations:

  • Requires Claude to change behavior. Claude must choose to use context_mode.execute() instead of built-in tools. If it forgets and uses Bash directly, no compression happens. This depends on prompt/CLAUDE.md instructions.
  • Generic extraction. The sandbox captures stdout, but doesn't understand what's inside. A screenshot (base64) and a JSON API response get the same treatment. No OCR, no content-aware processing.
  • Screenshots remain unsolved. Playwright MCP returns screenshots as tool_result content — Context Mode can't intercept this because it's not a proxy for Playwright. It's a separate tool Claude has to explicitly choose to use.
  • Two-hop latency. Each query requires Claude to formulate a sandbox command, wait for execution, then formulate the next. More round-trips than direct tool use.
  • Doesn't compress existing context. Only works for new tool calls. Old screenshot/DOM data already in context stays there.

token-optimizer-mcp

Open Source github.com/ooples/token-optimizer-mcp

Claims 95%+ token reduction through caching, compression, and "smart tool intelligence." Combines multiple strategies: deduplication, response caching, and output truncation.

Limitations:

  • Caching helps with repeated queries but doesn't solve novel large outputs
  • No content-aware compression (no OCR, no DOM cleanup)
  • Truncation risks losing important data at the end of long outputs

Playwright CLI (Alternative to Playwright MCP)

Tool testcollab.com

Instead of streaming accessibility trees and screenshot bytes into context, saves everything to disk. The agent reads files on-demand. Dramatically lower token consumption by design.

Limitations:

  • Fundamentally changes the workflow — not a drop-in replacement for Playwright MCP
  • Agent must explicitly decide to read files, adding extra turns
  • Screenshots saved to disk are still base64 when read — no OCR compression

@contextstream/mcp-server

npm

Consolidated domain tools with ~75% token reduction. Merges related MCP tools into fewer, more efficient interfaces.

Targets: Hybrid — reduces both definitions and some output overhead through consolidation.

Critique of Existing Solutions Analysis

  • Input vs Output split may be false dichotomy. We present these as two separate problems, but they interact: reducing input bloat (tool definitions) frees space for output bloat (tool results) to accumulate even more before compact triggers. Solving one may worsen the perceived severity of the other.
  • Competitor claims are unverified. claude-context-mode claims "98% reduction" and token-optimizer-mcp claims "95%+." We haven't independently tested any of these tools. Their benchmarks may use favorable test cases or different metrics than ours.
  • Gap analysis assumes OCR is the answer. We frame "no OCR for screenshots" as the critical gap. But there are alternative approaches: (1) Playwright could return smaller images; (2) Screenshots could be replaced with descriptions generated by a vision model (more expensive but higher quality); (3) Claude could simply not take screenshots as often if instructed properly.
  • "No tool does X" ≠ "X is needed." Maybe screenshot OCR doesn't exist because (a) Playwright sessions are a minority use case, (b) users who hit this problem simply use fewer screenshots, or (c) the solution is to fix Playwright MCP, not add a proxy layer.
  • Solution complexity. Each additional tool (MCP proxy, OCR engine, Gemini API) adds failure modes, latency, and maintenance burden. The "simplest fix" for most users may be /compact + shorter sessions, not more infrastructure.

6. Community Experience: The Compaction Quality Problem

Our web research across Reddit, GitHub issues, blog posts, and developer forums revealed a consistent pattern: compaction is widely perceived as lossy and harmful.

"Claude Saves Tokens, Forgets Everything"

Alexander Golev documented how project instructions were followed perfectly before compaction and violated 100% of the time after. He calls it "soft degradation that erases your working context invisibly."

DoltHub: "Code is definitely dumber after compaction"

DoltHub's analysis: "The code is definitely dumber after compaction — it doesn't know what files it was looking at and needs to re-read them, and will make mistakes that were specifically corrected earlier in the session." Estimated only 20-30% of original detail is retained.

The Auto-Compact Buffer Debate

Claude Code reserves a 33-45k token buffer (16-22% of context) for auto-compaction. Community is divided:

  • Melvynx: "Disable auto compact. You're losing 33% of your context window for nothing."
  • SFEIR Institute: "Run /compact at 70% capacity. Break work into 30-minute sprints."
  • Env var: CLAUDE_AUTOCOMPACT_PCT_OVERRIDE controls when auto-compaction fires

Source: Claude Code Context Buffer: The 33K-45K Token Problem

6.1. What the Community Does to Survive

The "Three Files" Method

Multiple developers independently converge on maintaining persistent state files:

  • plan.md — current task plan and objectives
  • context.md — key decisions, discoveries, constraints
  • tasks.md — checklist of remaining work

After compaction or session restart, "continue" causes Claude to read these docs, restoring context. This is a manual workaround for information loss.

Source: How I Stopped Claude Code From Losing Context After Every Compaction

Anthropic's Own Context Editing (Sept 2025)

Anthropic acknowledged the problem and released "context editing" + "memory tool" — achieving 84% token reduction in a 100-turn web search evaluation with 39% performance improvement over baseline.

Context editing automatically clears stale tool calls while preserving conversation flow. This validates the core premise of our approach.

6.3. The "Context Rot" Research

Context Rot (Chroma Research)

Formal research showing LLM performance degrades as context fills. Models favor tokens at the start and end, losing middle context ("Lost in the Middle" effect, Liu et al 2023). This means the worst place for important information is the middle of a long conversation — exactly where it ends up after many turns.

Implication for Context Trash: by compressing old middle content while preserving recent and initial context, we align with the model's natural attention patterns.

Critique of Community Evidence

  • Survivorship bias. People who write blog posts about compaction problems are the ones who were frustrated. Developers for whom compaction works fine don't write posts about it. We're sampling from a biased distribution of negative experiences.
  • "20-30% detail retained" is one blog post's estimate. This number from DoltHub is widely cited but wasn't rigorously measured. It's a subjective impression, not a quantitative analysis. Actual retention may vary significantly by session type.
  • Context Rot has nuance. The "Lost in the Middle" effect is real but varies by model and task. Claude's architecture may handle mid-context information differently than the models studied in Liu et al 2023. We cite it as supporting evidence but haven't verified it applies to Claude specifically.
  • Community workarounds work. The "Three Files" method (plan.md, context.md, tasks.md) is free, requires no tooling, and is widely successful. This undermines the case for complex compression infrastructure — maybe the real solution is better session discipline, not more technology.
  • Anthropic's context editing (84% reduction) may make all of this moot. If Anthropic ships better built-in context management, third-party compression tools become unnecessary. We're analyzing a problem that the platform vendor is actively solving.

7. Potential Direction: Content-Aware Element-Level Compression

The gap analysis in section 5.6 points toward a strategy that no existing tool implements: selective, content-type-aware compression of individual tool_result elements, rather than full conversation summarization or uniform masking.

Core Idea

Instead of summarizing the entire conversation (compact) or blindly masking old outputs (observation masking), identify individual "fat" content blocks and replace each with a semantically equivalent but dramatically smaller representation. The conversation structure is preserved — messages stay in order, small elements remain untouched, and the compression method is tailored to the content type.

7.1. Three Compression Tiers

Tier 1: Screenshot → OCR → Markdown (300x+ compression)

Base64 PNG images in tool_results are decoded and processed through OCR (tesseract or Apple Vision Framework). The extracted text is formatted as Markdown, preserving UI structure, table data, button labels, and text content.

7.2. Scoring Algorithm

Not all fat elements should be compressed. The scoring algorithm determines priority:

squeeze_score = size_weight × type_weight × (1 - recency_weight)

Where:
  size_weight    = tokens / 1000          // bigger = higher priority
  type_weight    = screenshot: 1.0        // always compress
                   image: 0.9
                   dom_snapshot: 0.7
                   file_read: 0.5
                   tool_output: 0.4
  recency_weight = last 3 msgs: 1.0       // never compress
                   msgs 4-10: 0.7
                   msgs 11-20: 0.3
                   msgs 20+: 0.0          // always eligible

Selection: sort by score DESC, accumulate savings until target reached.

Context %ActionTarget Savings
<70%No action-
70-80%Gentle squeeze: screenshots onlyFree 20% of context
80-90%Standard squeeze: screenshots + DOM + large textFree 30%
>90%Aggressive squeeze: everything >1k tokens older than 5 msgsFree 40%+

Critique of the Element-Level Compression Approach

  • JSONL rewriting is dangerous. Claude Code's JSONL files are append-only by design. Rewriting them (even with .bak backup) risks corrupting session state, breaking --resume, or causing data loss. This is the most fragile point of the entire approach.
  • Scoring algorithm is handcrafted. The recency weights, type weights, and thresholds (70%, 80%, 90%) are educated guesses, not empirically validated. Wrong thresholds could compress too aggressively (losing useful context) or too conservatively (not freeing enough space).
  • OCR loses visual information. For CSS debugging, responsive design checking, or visual regression testing, the spatial layout, colors, and element positioning matter. OCR throws all of this away. A developer saying "the button is overlapping the header" needs the visual, not the text.
  • Latency cost. OCR (1-5s per image) + Gemini API (3-5s per text block) adds up. A session with 10 screenshots at 80% capacity would add 10-50 seconds of processing before Claude even starts responding. This may be more disruptive than just letting compact fire.
  • MCP proxy adds complexity. Instead of one MCP server, the user now has a proxy wrapping an MCP server. Debugging failures (is the bug in the proxy? the MCP server? the connection?) becomes harder. Error messages may be obscured.
  • Gemini dependency for Tier 3. Relying on a competing AI company's API for text summarization introduces: (a) cost, (b) latency, (c) privacy concerns (code sent to Google), (d) availability risk. Many users won't want their source code processed by Gemini.
  • PostToolUse hooks may eventually support modification. GitHub #18594 requests writable hooks. If Anthropic implements this, the entire Proxy MCP architecture becomes unnecessary — a simple hook script would suffice.

7.3. What OCR Captures vs. Loses

Captured (useful)Lost (acceptable)
  • All visible text (headings, paragraphs, labels)
  • Button labels and menu items

Key insight: For most coding tasks (debugging UI, reading data, understanding page structure), the OCR output contains all the information that was actually useful from the screenshot. Visual layout information is rarely referenced in subsequent conversation turns.

OCR Limitations & Risks

  • OCR tested on only 2 screenshots. Our 300-500x compression ratio comes from testing on a Google Ads dashboard and a session manager UI. Both are text-heavy UIs. Screenshots of charts, graphs, maps, video frames, or design mockups would yield much worse OCR results or completely useless output.
  • macOS dependency. Apple Vision Framework (our best OCR engine) only works on macOS. Linux and Windows users need Tesseract (lower quality) or Gemini Vision (API cost + privacy concern). This fragments the user experience.
  • Non-English OCR quality varies. While both engines claim multilingual support, OCR quality degrades significantly for CJK scripts, RTL languages, and mixed-language UIs. Our testing was limited to English + some Russian.
  • Claude is multimodal. Claude can natively understand images. By replacing a screenshot with OCR text, we're trading Claude's own visual understanding (which may notice layout issues, alignment problems, visual bugs) for a lower-fidelity text-only representation. In some cases, Claude would make better decisions with the raw image.
  • Compression ratio ≠ information retention. 500x compression sounds impressive, but it means we're keeping 0.2% of the original data. Even if 99.8% was base64 encoding overhead, the spatial relationships between elements carry information that pure text extraction misses.

8.4. Processing Pipeline Detail

async function smartSqueeze(sessionId: string, jsonlPath: string) {
  // 1. Read last assistant message's input_tokens
  const currentTokens = getLastInputTokens(jsonlPath);
  const threshold = 140000; // 70% of 200k

  if (currentTokens < threshold) return; // no action needed

  // 2. Parse JSONL, identify fat elements
  const elements = parseFatElements(jsonlPath, minTokens: 1000);

  // 3. Score each element
  const scored = elements.map(el => ({
    ...el,
    score: (el.tokens / 1000) * typeWeight(el.type) * (1 - recencyWeight(el.msgIndex))
  }));

  // 4. Select targets (accumulate until savings target met)
  const targetSavings = (currentTokens - 100000); // bring down to 50%
  const targets = selectTargets(scored, targetSavings);

  // 5. Process each target
  for (const target of targets) {
    if (target.type === 'base64_image') {
      target.replacement = await ocrToMarkdown(target.imageData);
    } else if (target.type === 'dom_snapshot') {
      target.replacement = cleanAccessibilityTree(target.text);
    } else if (target.type === 'large_text' && target.tokens > 2000) {
      target.replacement = await geminiSummarize(target.text);
    }
  }

  // 6. Rewrite JSONL with replacements
  await rewriteJsonl(jsonlPath, targets);
}

9.6. Privacy & Data Handling

Context compression inherently involves processing potentially sensitive content — screenshots of dashboards, source code, API responses. This is a critical consideration.

Data flow analysis by compression tier

TierWhere data is processedWhat leaves your machinePrivacy risk
Tier 1: OCR
(screenshots)		 100% local — Apple Vision / Tesseract run on device Nothing None
Tier 2: Regex
(DOM snapshots)		 100% local — regex string replacement Nothing None
Tier 3: AI Summary
(large text)		 Remote API — Gemini Flash (user's own key) Text content sent to Google Medium — depends on content

Privacy-first design principles

  • Tiers 1 & 2 are fully local. Screenshots and DOM snapshots — the two biggest context consumers (90%+ of savings) — never leave your machine.
  • Tier 3 is opt-in. AI summarization of text content can be disabled entirely. Without it, you still get 80-90% of the benefit from OCR + regex alone.
  • BYOK (Bring Your Own Key). If Tier 3 is enabled, the user provides their own Gemini API key. No data routes through a third-party server.
  • No telemetry, no cloud. The MCP proxy runs as a local process. No data is collected, logged externally, or sent anywhere except the optional Gemini API call.
  • Original data preserved. Compressed screenshots are saved locally before replacement. Nothing is irreversibly destroyed.

What about the data already going to Anthropic?

Important context: all tool results already go to Claude's API (Anthropic's servers) as part of the conversation. The full 250k-token base64 screenshot is already being sent to Anthropic on every turn. Context Trash actually reduces data exposure by replacing raw screenshots with text-only OCR output before it reaches Claude.

In other words: without compression, Anthropic sees your full screenshots. With compression, Anthropic sees only the extracted text. Privacy improves, not degrades.

Key Differentiators of This Approach

  1. Content-type awareness: Unlike observation masking (which discards everything) or compact (which summarizes everything), this approach would use the right compression method for each content type — OCR for images, regex for DOM, AI for text.
  2. Conversation structure preserved: Messages stay in place, small elements untouched, only bloated blocks are replaced. The agent sees a normal conversation, not a summary blob.
  3. Targets the #1 offender: Our data shows screenshots consume 90-98% of tokens in browser sessions. OCR provides 300-500x compression with near-zero information loss for coding tasks. No existing tool targets this.
  4. Complementary to built-in compaction: Could run before compact triggers, potentially preventing the need for compact entirely. If compact still fires, it operates on a much smaller context.

Overall Research Limitations & Honest Assessment

  • Confirmation bias throughout. This research was conducted with a predetermined hypothesis ("tool results are the problem, OCR is the solution"). We selected data, literature, and framing that supports this conclusion. A neutral analysis might reach different conclusions about the relative importance of context compression.
  • Single-developer dataset is not a study. N=1 developer, N=15 sessions. In academic terms, this is an anecdotal report, not an empirical study. Making generalizable claims from this data is scientifically questionable.
  • The simplest solution might be the best. Before building MCP proxies with OCR pipelines, consider: (1) Start new sessions more often; (2) Use /compact proactively; (3) Configure CLAUDE.md to remind Claude about session discipline; (4) Use Playwright CLI instead of MCP. These require zero infrastructure.
  • Market timing risk. Anthropic is actively improving context management. Claude Code v3+ may ship with built-in screenshot compression, smarter compaction, or significantly larger context windows. Any third-party tool built today may be obsoleted within months.
  • We tested no solution end-to-end. This research proposes approaches but implements none. The compression ratios are theoretical (based on OCR tests on 2 images) and the architecture is conceptual. Actual implementation may reveal fundamental issues we haven't anticipated.
  • Cost-benefit analysis is incomplete. We detail the benefits of compression (longer sessions, fewer compacts) but don't quantify the costs: development time, maintenance overhead, latency per turn, debugging complexity, and the cognitive load of managing yet another tool in the stack.
