Context Trash MCP

Proxy MCP-сервер для сжатия тяжёлого контента перед попаданием в контекст Claude Code. Пошаговая архитектура, стоимость, и кто за что платит.

Главный вопрос: что это и для кого

В одном предложении

Context Trash MCP — это прокси, который садится между Claude Code и любым MCP-сервером (Playwright, файлы, API и т.д.) и автоматически сжимает "жирные" ответы перед тем, как Claude их увидит.

Без Context Trash

Claude Code
  → вызывает Playwright "сделай скриншот"
  → Playwright возвращает base64 PNG
  → 250,000 токенов попадают в контекст
  → контекст забит за 3 скриншота
  → compact → потеря памяти

Со Context Trash

Claude Code
  → вызывает Playwright "сделай скриншот"
  → Playwright возвращает base64 PNG
  → Context Trash перехватывает
  → OCR: извлекает текст → 500 токенов
  → Claude видит текст, не картинку
  → контекст чистый часами

Итого: 90% функциональности — бесплатно

OCR скриншотов (главный use case, 300x сжатие) = бесплатно, работает локально.

Чистка DOM snapshots = бесплатно, regex.

Gemini нужен только для "умной" суммаризации текстовых блоков. Без него тоже работает — просто обрезаем до N символов вместо суммаризации.

Gemini Flash стоит копейки: $0.075 за 1M входных токенов. Одна сессия с 50 элементами по 5k = 250k токенов = $0.02. Два цента.

Пошаговый сценарий: что происходит

Сценарий 1: Playwright скриншот (самый частый)

1
Юзер → Claude Code

"Открой страницу example.com и покажи что там"

~20 токенов

↓
2
Claude Code → Context Trash MCP

Claude вызывает tool browser_navigate + browser_take_screenshot

Context Trash получает вызов и проксирует его дальше к настоящему Playwright MCP

Context Trash тут просто пересылает запрос, ничего не меняет

↓
3
Playwright MCP → ответ

Playwright открывает страницу, делает скриншот, возвращает:

{ type: "image", source: { type: "base64", data: "iVBORw0KGgo..." } }
// 300KB base64 = ~100,000 токенов

~100,000 токенов base64

↓
4
Context Trash — перехват и обработка

Вот тут магия:

  1. Видит что ответ содержит base64 image
  2. Декодирует base64 → PNG файл → сохраняет на диск (для истории)
  3. Запускает OCR (tesseract или Apple Vision) → получает текст
  4. Заменяет base64 image на текст:
{ type: "text", text: "[Screenshot OCR — example.com]\n
## AI Recruiter — Hire Top Talent 10x Faster\n
- Navigation: Products, Pricing, Resources, Login\n
- Hero: 'Your ATS gathers thousands of profiles...'\n
- CTA: 'Book a demo' button\n
- Stats: 10x faster, 90% accuracy" }

100,000 ток (было) → ~200 ток (стало)

$0 — tesseract работает локально

↓
5
Claude Code получает сжатый ответ

Claude видит текстовое описание страницы вместо 100k base64.

Отвечает юзеру: "На example.com вижу landing page с заголовком 'AI Recruiter'..."

Claude НЕ ЗНАЕТ что вместо картинки получил текст. Для него это просто tool_result.

Сценарий 2: DOM snapshot (browser_snapshot)

1
Claude → browser_snapshot

Claude хочет увидеть структуру страницы для клика

↓
2
Playwright возвращает accessibility tree
### Open tabs\n- 0: [ExampleApp](https://example.app)\n- 1: ...\n### Snapshot\n
- generic [ref=e47]:\n  - generic:\n    - generic:\n      - generic: Desktop\n  - iframe [ref=e49]:\n    - generic [active] [ref=f71e1]: ...

~14,000 токенов

↓
3
Context Trash — regex cleanup

Убирает [ref=xxx], схлопывает пустые generic обёртки, оставляет семантику:

### Tabs: ExampleApp (current), Notion, Gmail\n### Page: https://example.app\n
- link "Solve your problem" → #book\n- heading[h3]: "Your ATS gathers thousands of profiles"\n
- button: "Book a demo"\n- section: Stats (10x, 90%)

14,000 ток → ~2,000 ток

$0 — regex, локально

Важно: ref-ы нужны для кликов! Context Trash сохраняет маппинг ref → element отдельно и подставляет ref обратно когда Claude делает click.

Сценарий 3: Большой текстовый результат (Read, Bash, Grep)

1
Claude → Read tool

Claude читает файл на 8000 токенов

↓
2
Результат: полное содержимое файла

~8,000 токенов

↓
3
Context Trash — решение

Два варианта (настраивается):

A) Без Gemini (бесплатно): Если файл > порога (напр. 4000 ток) и это НЕ последний запрос — обрезать до первых/последних 500 токенов + заметка "[truncated, full file: /path/to/file]"

B) С Gemini Flash (2 цента за сессию): Отправить файл в Gemini Flash с промптом "Summarize this code file in 3-5 bullet points" → получить summary 200 токенов

8,000 ток → ~500 ток

Вариант A: $0 | Вариант B: ~$0.0006

Как подключить (для юзера)

Шаг 1: Установить

npm install -g context-trash-mcp
# или
npx context-trash-mcp setup

Шаг 2: Настроить в ~/.claude.json

Было (напрямую к Playwright):

{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@anthropic/mcp-playwright"]
    }
  }
}

Стало (через Context Trash):

{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["context-trash-mcp",
               "--wrap", "npx @anthropic/mcp-playwright",
               "--ocr", "tesseract",
               "--max-tokens", "2000"]
    }
  }
}

Шаг 3 (опционально): Подключить Gemini для суммаризации текста

{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["context-trash-mcp",
               "--wrap", "npx @anthropic/mcp-playwright",
               "--ocr", "tesseract",
               "--max-tokens", "2000",
               "--gemini-key", "env:GEMINI_API_KEY",
               "--summarize-above", "4000"]
    }
  }
}

Или через env переменную:

export GEMINI_API_KEY=your-key-here
# Gemini Flash: бесплатный tier = 15 запросов/мин, 1M токенов/день
# Платный: $0.075/1M input tokens

Когда Claude видит картинку, а когда нет

Важный нюанс: Claude умеет видеть картинки!

Claude — мультимодальная модель. Он может анализировать изображения напрямую. Но:

  • Каждая картинка = тысячи токенов. Один скриншот Playwright — 100-400k токенов.
  • Для большинства задач текст достаточен. Если Claude проверяет "есть ли кнопка Login на странице" — ему не нужна картинка, достаточно OCR-текста "button: Login".
  • Context Trash сохраняет картинку на диск. Если Claude действительно нужно увидеть визуал (проверить цвет, layout) — он может прочитать файл.

Конфигурация: что сжимать, а что нет

context-trash-mcp \
  --wrap "npx @anthropic/mcp-playwright" \
  --ocr tesseract \                    # OCR engine
  --max-tokens 2000 \                  # порог для сжатия текста
  --save-originals ~/.squeeze/images \ # куда сохранять оригиналы
  --keep-images false \                # false = OCR, true = пропускать картинки как есть
  --clean-snapshots true \             # чистить DOM snapshots
  --gemini-key env:GEMINI_API_KEY \    # опционально
  --summarize-above 4000              # суммаризировать тексты > 4k tok

Режимы:

РежимЧто делаетКогда использовать
--keep-images false Все картинки → OCR → текст Обычная разработка, чтение страниц
--keep-images true Картинки проходят как есть Визуальное тестирование, дизайн-ревью
--keep-images recent Последний скриншот — как есть, старые → OCR Баланс: Claude видит текущее состояние, но старые не жрут контекст

FAQ

Нужно ли юзеру что-то платить?

Нет. Базовая функциональность (OCR + regex) полностью бесплатна и локальна. Gemini — опция, и даже она стоит 2 цента за сессию.

Нужно ли юзеру прокидывать свои ключи?

Только если хочет Gemini-суммаризацию. По умолчанию: --keep-images false --ocr tesseract — ноль зависимостей кроме tesseract.

Будет ли Claude хуже работать без картинок?

Для 95% задач — нет. OCR захватывает все тексты, кнопки, таблицы, данные. Claude не нужна картинка чтобы понять "на странице есть кнопка Login". Для visual testing (цвета, layout) — можно включить --keep-images true.

А если Claude нужно кликнуть на элемент?

DOM snapshots сохраняют ref-ы! Context Trash убирает мусор но маппинг ref → element хранится. Когда Claude делает browser_click(ref="e47") — Context Trash пересылает как есть.

Это только для Playwright?

Нет. Context Trash — generic proxy для любого MCP. Playwright — главный use case (300x сжатие), но он также сжимает Read, Bash, WebFetch, Grep и любой другой tool с большим output.

Как это соотносится с claude-context-mode?

claude-context-mode — похожая идея, но: (1) не делает OCR, (2) требует sandbox для каждого языка, (3) generic extraction без понимания контента. Context Trash специализирован на типах контента и использует OCR для изображений.
